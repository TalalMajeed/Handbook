Step 2 · Header Stripping
Remove Garbage Text From Every Page
✓ Done ▼
Every page had a running header like "NUST Undergraduate Student Handbook 87" embedded in the text. Since this appeared on every single page, it was polluting the search index — making the words "nust", "undergraduate", "handbook" appear in every chunk and destroying their search value. We strip all these headers before doing anything else.
💡 Imagine if every page of a textbook had the title printed inside the paragraph — you'd have to remove that before indexing the actual content.
What we remove
"NUST * Student Handbook N"
Standalone page numbers
School name stamps
Why it matters
If every chunk has "nust student handbook", TF-IDF gives those words zero weight — they can't distinguish any chunk from another.

Step 3 · Chunking
Cut Text Into Meaningful Pieces
✓ Done ▼
We can't search the whole handbook at once. We cut it into small pieces called chunks, each 50–250 words. We use sentence-aware chunking — we accumulate complete sentences and only start a new chunk at a sentence boundary. One sentence overlaps between chunks so context is never lost.
💡 Like cutting a book into index cards — but you only cut between paragraphs, never mid-sentence, so each card always contains complete ideas.
Old method (broken)
300-word windows, 50% overlap — split rules mid-sentence so "CGPA is 2.75 for Engineering" ended up in two different chunks
New method (fixed)
Sentence-boundary cutting, 50–250 words, 1-sentence overlap — complete rules always stay in one chunk
~300+ chunks created 50–250 words each 1-sentence overlap

Step 4 · SimHash Deduplication
Remove Duplicate Chunks
✓ Done ▼
Some handbook sections repeat content. We use SimHash to fingerprint every chunk and remove near-duplicates before indexing them.
💡 Like a DNA test for text. Two chunks with very similar DNA (Hamming distance ≤ 3 bits) are considered duplicates — we keep only one.
1
For each chunk, tokenise into words
2
MD5 hash each token → 64-bit binary number
3
Build a 64-position accumulator: +1 if bit=1, -1 if bit=0
4
Fingerprint = sign of each accumulator position
5
Compare fingerprints: if Hamming distance ≤ 3 → duplicate → discard
Hamming distance = count of bit positions that differ
Threshold ≤ 3 bits → near-duplicate → remove
SimHash 64-bit fingerprint Hamming distance ≤ 3

Step 5 · Sentence Embeddings
Teach the Computer What Each Chunk Means
✓ Done ▼
Words alone don't capture meaning. We use a pre-trained AI model all-MiniLM-L6-v2 to convert each chunk into a list of 384 numbers called an embedding. Chunks with similar meanings get similar number-lists, even if they use completely different words.
💡 Like giving every chunk GPS coordinates in "meaning space". Chunks about attendance are near each other. Chunks about fees are near each other. Far from chunks about, say, sports.
Model used
all-MiniLM-L6-v2
(runs fully offline, no internet)
Output per chunk
384 numbers (a vector). Stored inside chunks.pkl for fast retrieval later.
384 dimensions L2 normalised semantic similarity

Step 6 · PageRank
Find the Most Important Chunks
✓ Done ▼
We build a graph where every chunk is a node. If two chunks are semantically similar (embeddings are close), we draw an edge between them. Then we run the PageRank algorithm — the same one Google uses to rank web pages — to find which chunks are most "central" to the handbook.
💡 Like Google ranking websites. A chunk connected to many other important chunks (core policy rules) ranks higher than an isolated edge-case chunk.
Similarity graph: edge if cosine(emb_i, emb_j) > 0.65
PageRank weight in final score: 0.02 (tiebreaker only)
Damping factor: 0.85
PageRank networkx library weight = 0.02

Step 7 · TF-IDF Index
Build the Keyword Search Engine
✓ Done ▼
TF-IDF (Term Frequency × Inverse Document Frequency) scores how important each word is to each chunk. Common words like "student" appear everywhere so they get low scores. Rare specific words like "CGPA" or "probation" appear in few chunks so they get high scores — making those chunks findable.
Improvements we made
Domain stop-words: removed "nust", "university", "student", "handbook" — they appear everywhere and are useless

Bigrams: index pairs like "credit hours", "grade point"

Synonym expansion: "GPA" → also indexes "cgpa grade point average"
Settings
15,000 max features
sublinear_tf = True (log scale)
max_df = 0.85 (drop words in 85%+ of chunks)
ngram_range = (1,2)
TF-IDF(word, chunk) = TF × IDF
TF = log(1 + count) [sublinear]
IDF = log(total_chunks / chunks_with_word)
TF-IDF 15k features bigrams synonym expansion

Step 8 · MinHash + LSH Index
Build the Fast Approximate Search Engine
✓ Done ▼
TF-IDF is exact but slow on huge datasets. MinHash + LSH is an approximate method that's much faster — it trades a small amount of accuracy for a big speed gain.
1
Shingles: Each chunk becomes a set of 3-word sequences ("minimum cgpa required", "cgpa required for", …)
2
MinHash: Apply 128 hash functions to the shingle set → get a 128-number "signature". Two similar chunks will have similar signatures.
3
LSH Banding: Split the 128 numbers into 16 groups of 8. Hash each group into a "bucket". If two chunks land in the same bucket for any group → they're candidates for being similar.
4
At query time: only compare against chunks that share a bucket. Much faster than comparing against all chunks.
128 hash functions ÷ 16 bands = 8 rows per band
Jaccard threshold s* = (1/16)^(1/8) ≈ 0.707
Collision probability at s*: 50%
MinHash 128 hash functions 16 bands s* ≈ 0.707

Step 4 · TF-IDF Pre-Filter
Fast Keyword Search → Top 40 Candidates
✓ Done ▼
The expanded query is converted to a TF-IDF vector and compared against all 300+ chunk vectors. Chunks scoring above 0.005 are kept as candidates — chunks with zero keyword overlap are excluded entirely.
⚠️ Bug we fixed: Without this gate, high-PageRank chunks about Islamic etiquette were appearing in GPA query results because they scored 0 on TF-IDF but had high PageRank. Gate prevents this.
cosine_similarity(query_vec, chunk_vec) for all chunks
Keep: score > MIN_TFIDF_SCORE (0.005) AND top 40
Discard: anything with zero keyword overlap
Input
300+ chunks
Output
Top 40 keyword-relevant candidates
exact baseline O(vocab × chunks) MIN_TFIDF = 0.005

Step 5 · Semantic Re-Ranking
Score by Meaning, Not Just Keywords
✓ Done ▼
The query is embedded using the sentence-transformer model (the original cleaned query, not the expanded one — synonym noise would hurt embedding quality). We compute dot-product similarity between the query embedding and each of the 40 candidate chunk embeddings.
💡 TF-IDF found chunks with matching words. Semantic scoring finds chunks with matching meaning — even if they use different words.
semantic_score = dot_product(query_embedding, chunk_embedding)
Both vectors are L2-normalised → result = cosine similarity
Range: -1 to +1 (closer to 1 = more similar meaning)
all-MiniLM-L6-v2 cosine similarity semantic weight = 0.28

Combine All Signals Into One Score
✓ Done ▼
Each of the 40 candidates gets a final hybrid score combining TF-IDF keyword match, semantic meaning similarity, and PageRank importance. We sort and return the top 8.
score = 0.70 × TF-IDF_cosine
      + 0.28 × Semantic_cosine
      + 0.02 × PageRank

Top 8 chunks returned as "Retrieved Evidence"
⚠️ Fix applied: PageRank was 0.15 (too high) — it was promoting irrelevant but popular chunks. Reduced to 0.02 so it's a tiebreaker only.

Step 7 · Groq Generation
Send Top Chunks to AI for a Clean Answer
✓ Done ▼
The top 5 retrieved chunks are sent to the Groq API (Llama 3.1 8B) along with the original question. The model reads only those chunks and forms a complete, accurate answer. It streams back word by word.
⚠️ Why not flan-t5-base? Our original local model (250M params) gave one-word answers and couldn't reproduce tables. Groq's Llama 3.1 8B is free, fast, and handles complex answers correctly.
